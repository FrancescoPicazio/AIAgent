# AI Software Engineer Agent - Parte 3

# Code Intelligence Layer: analisi codice, AST, Dependency Graph e Vector Database

## Obiettivo

Creare il livello di intelligenza del sistema che permette all'agente AI di comprendere un progetto software complesso.

L'agente non deve analizzare il codice come semplice testo.

Deve costruire una rappresentazione strutturata del progetto tramite:

* Abstract Syntax Tree (AST);
* analisi delle dipendenze;
* grafo delle chiamate;
* indicizzazione semantica;
* metadata delle funzioni e classi.

Questo livello permette di:

* trovare il codice corretto;
* valutare l'impatto delle modifiche;
* comprendere relazioni tra componenti;
* fornire contesto corretto agli agenti.

---

# Principio generale

Il codice sorgente viene trasformato in una rappresentazione multilivello.

Pipeline:

```text
Repository

    ↓

Code Parser

    ↓

AST Analysis

    ↓

Metadata Extraction

    ↓

Dependency Graph

    ↓

Vector Embedding

    ↓

Knowledge Base

    ↓

Agent Context
```

---

# 1. Code Scanner

## Responsabilità

Prima fase eseguita quando:

* viene creato un progetto;
* viene aperto un progetto esistente;
* viene modificato codice;
* viene effettuato un commit.

Il Code Scanner deve individuare:

* linguaggi utilizzati;
* struttura directory;
* file sorgente;
* configurazioni;
* framework;
* dipendenze.

Output:

```text
.ai/knowledge/project_scan.json
```

Esempio:

```json
{
    "languages": [
        "Python",
        "JavaScript"
    ],
    "frameworks": [
        "FastAPI",
        "React"
    ],
    "files": 542,
    "directories": 37
}
```

---

# 2. AST Parser

## Obiettivo

Comprendere la struttura interna del codice.

L'analisi deve essere basata sulla struttura sintattica, non sul testo.

Tecnologia proposta:

* Tree-sitter

Motivazione:

* supporto multi linguaggio;
* parsing veloce;
* struttura uniforme.

---

## Informazioni estratte

Per ogni file:

```text
File

 ├── Classes
 │
 ├── Functions
 │
 ├── Imports
 │
 ├── Variables
 │
 └── Decorators
```

---

Esempio:

File:

```python
auth.py
```

Output:

```json
{
 "file": "auth.py",

 "functions": [
    {
      "name": "login",
      "parameters": [
        "username",
        "password"
      ],
      "return": "Token"
    }
 ],

 "classes": [
    {
      "name": "AuthService"
    }
 ]
}
```

# 3. Function Metadata Extraction

Ogni funzione deve avere una scheda tecnica.

Formato:

```json
{
"name": "calculate_price",

"type": "function",

"description":
"Calcola il prezzo finale",

"inputs":
[
 {
  "name": "items",
  "type": "List"
 }
],

"output":
"float",

"dependencies":
[
 "DiscountService"
],

"complexity":
"medium",

"last_modified":
"task_042"
}
```

Queste informazioni alimentano:

* glossary.md;
* vector DB;
* dependency graph.

# 4. Dependency Graph

## Obiettivo

Creare una mappa delle relazioni software.

Il grafo rappresenta:

* chi chiama cosa;
* quali classi dipendono da altre;
* quali moduli sono collegati.

Esempio:

```text
OrderController

        |
        v

OrderService

        |
        +------------+
        |            |
        v            v

Database      PaymentService
```

---

## Utilizzo

Quando l'utente chiede:

"Aggiungi pagamento PayPal"

il sistema può analizzare:

```text
PaymentService

impatto:

- OrderService
- CheckoutController
- PaymentRepository
- Tests
```

# 5. Impact Analysis Engine

Prima di modificare codice l'agente deve stimare l'impatto.

Input:

```text
Modifica:
Cambiare User model
```

Analisi:

```json
{
 "affected_files": 18,

 "affected_functions": 43,

 "risk": "HIGH",

 "dependencies": [
    "Authentication",
    "Database",
    "API"
 ]
}
```

Questo report viene mostrato all'utente prima dell'approvazione.

# 6. Vector Database

## Obiettivo

Permettere ricerca semantica.

Il Vector DB non sostituisce il Dependency Graph.

Sono complementari.

---

## Dependency Graph risponde:

"Dove è collegato?"

Esempio:

```text
Se modifico login(), cosa cambia?
```

---

## Vector DB risponde:

"Dove esiste qualcosa di simile?"

Esempio:

```text
Dove viene gestita l'autenticazione?
```

---

# Chunking del codice

Il codice non deve essere diviso arbitrariamente.

Da evitare:

```text
Ogni 500 caratteri
```

Preferire:

```text
Un chunk = unità logica
```

Esempio:

```text
Classe

oppure

Funzione

oppure

Modulo
```

Ogni embedding deve contenere:

```json
{
"type": "function",

"name":
"authenticate",

"file":
"auth/service.py",

"context":
"Gestione autenticazione utenti",

"code":
"..."
}
```

# 7. Retrieval Engine

Quando un agente riceve un task:

Esempio:

"Implementa login OAuth"

non riceve tutto il repository.

Pipeline:

```text
Task

↓

Planner

↓

Retrieval

↓

Vector Search

↓

Graph Analysis

↓

Context Builder

↓

LLM
```

---

Il contesto finale:

```text
File rilevanti:

auth/service.py
user/model.py
database/user_repository.py

Funzioni:

authenticate()
create_user()
save_user()

Decisioni:

JWT già utilizzato.
Repository pattern obbligatorio.
```

---

# 8. Aggiornamento automatico

Dopo una modifica completata:

```text
Code Change

↓

AST Rebuild

↓

Update Graph

↓

Update Metadata

↓

Update Glossary

↓

Update Vector DB
```

L'indice deve sempre rappresentare lo stato reale del codice.

---

# 9. Controllo consistenza

Il sistema deve verificare:

* funzioni presenti ma non indicizzate;
* documentazione obsoleta;
* dipendenze cambiate;
* file modificati manualmente.

Esempio:

```text
Warning:

La funzione UserService.login()
è cambiata ma glossary.md non è aggiornato.

Aggiornare?
```

# Principio fondamentale

L'agente non deve "leggere codice".

Deve interrogare una rappresentazione intelligente del codice.

Il codice sorgente è la fonte primaria.

AST, Graph e Vector DB sono la memoria cognitiva che permette all'agente di ragionare sul progetto.
