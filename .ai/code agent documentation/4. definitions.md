# AI Software Engineer Agent - Parte 4

# Multi-Agent System con LangGraph: ruoli, stato e workflow

## Obiettivo

Definire il sistema multi-agente che permette all'AI Software Engineer Agent di lavorare come un team di sviluppo software.

L'architettura utilizza:

* LangGraph come orchestratore principale;
* LangChain per tool, prompt e integrazioni;
* Ollama come runtime locale dei modelli;
* LangSmith per osservabilità e valutazione.

Ogni agente ha una responsabilità specifica.

Nessun agente deve ricoprire tutti i ruoli contemporaneamente.

---

# Principio architetturale

Il sistema segue un modello:

```text id="8z1u0j"
Requirement

    ↓

Architect

    ↓

Planner

    ↓

Executor

    ↓

Reviewer

    ↓

Tester

    ↓

Memory Update

    ↓

Completion
```

Ogni fase produce un output strutturato utilizzato dalla fase successiva.

---

# Stato condiviso LangGraph

LangGraph utilizza uno stato comune che viene passato tra i nodi.

Esempio:

```python
AgentState:
    
    project_context

    user_request

    architecture_plan

    task_list

    current_task

    retrieved_context

    code_changes

    test_results

    review_results

    approval_status

    memory_updates
```

Lo stato rappresenta la memoria temporanea del workflow.

---

# Agent 1 - Architect Agent

## Responsabilità

L'Architect Agent analizza il requisito dal punto di vista progettuale.

Non scrive codice.

Il suo compito è decidere:

* quale approccio utilizzare;
* quali componenti coinvolgere;
* quali pattern applicare;
* quali rischi esistono.

---

## Input

Esempio:

```text
Aggiungere autenticazione OAuth Google
```

---

## Analisi

L'agente consulta:

* project.md;
* architecture.md;
* rules.md;
* dependency graph;
* ADR esistenti.

---

## Output

Esempio:

```yaml
architecture_decision:

approach:
  OAuth2 integration

affected_components:

- Authentication module
- User service
- Database layer

pattern:
  Adapter Pattern

risk:
  medium
```

---

# Agent 2 - Planner Agent

## Responsabilità

Trasforma la soluzione architetturale in task eseguibili.

Il planner non implementa.

---

## Input

Riceve:

* richiesta utente;
* decisione architetturale;
* contesto progetto.

---

## Output

Genera una task list.

Esempio:

```yaml
tasks:

- id: 1
  title:
    Create OAuth provider interface

  files:
    - auth/provider.py

  dependencies:
    - user_model


- id: 2
  title:
    Implement Google OAuth adapter

  files:
    - auth/google.py


- id: 3
  title:
    Add authentication tests

  files:
    - tests/test_auth.py
```

---

# Impact Report

Prima dell'esecuzione viene prodotto:

```yaml
impact:

files:
  8

functions:
  21

risk:
  medium

architecture_changes:
  false
```

Questo viene mostrato all'utente.

# Agent 3 - Coding Agent

## Responsabilità

Implementa il task assegnato.

Non decide:

* architettura;
* requisiti;
* priorità.

---

## Input

Riceve:

* task;
* file interessati;
* contesto recuperato;
* regole progetto.

---

## Prima fase

Analizza:

```text
File corrente

↓

Funzione target

↓

Dipendenze

↓

Pattern esistenti
```

---

## Output

Produce:

```json
{
"modified_files":

[
 "auth/service.py",
 "tests/test_auth.py"
],

"changes":

[
 "Added OAuthService",
 "Added login endpoint"
]
}
```

---

# Modifica tramite Patch

L'agente non riscrive file interi.

Produce modifiche atomiche:

```diff
- old code

+ new code
```

Vantaggi:

* rollback;
* controllo;
* minore rischio.

---

# Agent 4 - Reviewer Agent

## Responsabilità

Analizza la modifica.

Controlla:

* qualità;
* rispetto regole;
* sicurezza;
* mantenibilità;
* coerenza architetturale.

---

## Input

Riceve:

* codice modificato;
* diff;
* rules.md;
* architecture.md.

---

## Output

Esempio:

```yaml
review:

quality_score:
  8.5

issues:

- Missing error handling

suggestions:

- Add validation layer

status:
  NEEDS_FIX
```

# Correction Loop

Il sistema deve essere iterativo.

Workflow:

```text
             Coding Agent
                  |
                  v
              Reviewer
                  |
          +-------+-------+
          |               |
          v               v

       OK              Problems

          |               |

          v               v

       Tester        Coding Agent

                          |
                          v

                     Retry
```

---

# Limiti del loop

Per evitare cicli infiniti:

```python
MAX_RETRIES = 5
```

Dopo il limite:

```text
Task Failed

↓

Human Intervention
```

# Osservabilità LangSmith

Ogni nodo deve produrre trace.

Monitorare:

* input;
* output;
* token utilizzati;
* durata;
* errori;
* qualità risposta.

Metriche principali:

```text
Task Success Rate

Average Retry Count

Test Pass Rate

Human Approval Rate

Code Quality Score

Regression Rate
```

---

# Principio finale

Ogni agente deve avere un ruolo limitato.

La qualità nasce dalla collaborazione:

* Architect decide;
* Planner organizza;
* Coder implementa;
* Reviewer critica;
* Tester verifica;
* Memory mantiene conoscenza.

Il sistema non deve cercare di sostituire un programmatore in un singolo passaggio, ma replicare un ciclo professionale di sviluppo software.
