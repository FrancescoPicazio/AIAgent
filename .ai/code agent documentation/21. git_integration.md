# AI Software Engineer Agent - Parte 21

# Git Integration e Software Delivery Pipeline: versionamento, branch, PR e lifecycle del codice

## Obiettivo

Definire come l'agente interagisce con Git e con il ciclo completo di sviluppo software.

L'agente non deve semplicemente modificare file.

Deve comportarsi come un membro di un team reale:

* crea branch;
* prepara modifiche isolate;
* genera commit comprensibili;
* verifica il codice;
* prepara una review;
* mantiene storico delle decisioni.

---

# 1. Git Manager Agent

## Responsabilità

Gestisce tutte le operazioni Git.

Non deve essere il Coder Agent a chiamare direttamente Git.

Separazione:

```text id="x5m9p1"

Coder:

"Ho creato modifiche"


Git Agent:

"Creo branch e commit"

```

# 2. Branch Strategy

Ogni task parte da un branch isolato.

Esempio:

Richiesta:

```text id="p4n8m2"
"Aggiungi autenticazione OAuth"
```

---

Creazione:

```bash id="w7m3q9"
git checkout -b ai/feature/oauth-authentication
```

---

Struttura:

```text id="k8m2v5"

main

 |

 +-- ai/task-001-auth

 |

 +-- ai/task-002-payment

```

# 4. Intelligent Commit Generation

L'agente non deve fare:

```bash
git commit -m "changes"
```

---

Deve generare commit semantici.

Esempio:

```text id="z3m8q5"

feat(auth):

add OAuth2 login provider


Changes:

- added Google OAuth client
- added token validation
- updated auth tests

```

---

Formato:

Conventional Commits.

```text id="m5q8x1"

feat

fix

refactor

test

docs

security
```

---

# 5. Automatic Commit Splitting

Problema:

Un agente potrebbe modificare:

```text id="q7m2p9"

50 file

↓

1 commit enorme
```

---

Soluzione:

Dividere per responsabilità.

Esempio:

```text id="h4m8n2"

Commit 1:

Add authentication service


Commit 2:

Add database migration


Commit 3:

Add tests

```

# 9. Merge Policy

Non tutto può essere mergiato automaticamente.

---

## Auto merge


## Human approval

# Principio finale

Il codice generato dall'agente non deve essere solo corretto.

Deve essere:

* tracciabile;
* revisionabile;
* reversibile;
* comprensibile nel tempo.

Un vero software engineer non lascia solo codice.

Lascia una storia tecnica del perché quel codice esiste.
