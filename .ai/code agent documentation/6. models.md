# AI Software Engineer Agent - Parte 6

# AI Model Layer: Ollama, Model Routing e Prompt Architecture

## Obiettivo

Definire il livello dei modelli AI utilizzati dal sistema e la logica con cui gli agenti scelgono quale modello utilizzare.

Il sistema non deve utilizzare un singolo modello per ogni operazione.

Ogni agente deve utilizzare il modello più adatto al proprio compito.

Principio:

```text
Task diverso

↓

Modello specializzato

↓

Migliore qualità + minore consumo
```

---

# Local AI Runtime

## Ollama

Ollama è il runtime principale per l'esecuzione locale dei modelli.

Responsabilità:

* caricamento modelli;
* gestione memoria GPU;
* inferenza locale;
* esposizione API;
* gestione quantizzazione.

Architettura:

```text
LangGraph

    |

LangChain Ollama Wrapper

    |

Ollama Runtime

    |

Local GPU / CPU

    |

LLM
```

---

# Prompt Architecture

I prompt non devono essere hardcoded nel codice.

Devono essere parte del progetto.

Struttura:

```text
.ai/

└── prompts/

    ├── architect.md
    ├── planner.md
    ├── coder.md
    ├── reviewer.md
    └── tester.md
```

---

# Architect Prompt

Obiettivo:

Fornire decisioni tecniche.

Regole:

```markdown
Sei un Software Architect.

Non scrivere codice.

Analizza:

- requisiti;
- architettura esistente;
- vincoli.

Produci:

- soluzione proposta;
- alternative;
- rischi;
- impatto.
```

---

# Planner Prompt

Obiettivo:

Trasformare una richiesta in task.

Regole:

```markdown
Sei un Technical Planner.

Dividi il lavoro in task atomici.

Ogni task deve contenere:

- obiettivo;
- file coinvolti;
- dipendenze;
- rischio;
- criterio di completamento.
```

---

# Coding Agent Prompt

Obiettivo:

Modificare codice.

Regole:

```markdown
Sei un Senior Developer.

Prima di modificare:

1. analizza codice esistente;
2. verifica dipendenze;
3. rispetta rules.md.

Non creare soluzioni nuove se esistono pattern già presenti.
```

---

# Reviewer Prompt

Obiettivo:

Critica tecnica.

Regole:

```markdown
Analizza la modifica.

Valuta:

- qualità;
- sicurezza;
- manutenibilità;
- rispetto architettura.

Non accettare codice solo perché funziona.
```

---

# Tester Prompt

Obiettivo:

Analizzare fallimenti.

Regole:

```markdown
Analizza gli errori.

Determina:

- causa principale;
- file coinvolti;
- correzione proposta.

Non modificare codice direttamente.
```

---

# Structured Output

Ogni agente deve produrre output strutturati.

Non:

```text
"Secondo me bisogna modificare..."
```

Ma:

```json
{
"decision":
"Use Repository Pattern",

"reason":
"Reduce coupling",

"impact":
"medium"
}
```

Vantaggi:

* parsing automatico;
* controllo;
* passaggio dati tra agenti.

# Context Management

Il modello non deve ricevere tutto il progetto.

Ogni chiamata riceve:

```text
System Prompt

+

Project Rules

+

Relevant Memory

+

Retrieved Code

+

Current Task
```

---

Esempio:

```text
TASK:

Aggiungere OAuth


CONTEXT:

architecture.md

auth module

User model

OAuth ADR


RULES:

Repository pattern

Testing required
```

Small context

Search more

Expand only if needed

# Principio finale

L'agente non deve essere un singolo modello enorme.

Deve essere un sistema dove:

```text
Modello giusto

+

Prompt giusto

+

Contesto giusto

+

Tool giusti

=

Software Engineer Agent
```

La qualità nasce dall'orchestrazione, non solo dalla dimensione del modello.
