# AI Software Engineer Agent - Parte 2

# Struttura del progetto e sistema di memoria `.ai`

## Obiettivo

Definire la struttura interna che permette all'agente AI di comprendere, mantenere e modificare un progetto software nel tempo.

La conoscenza dell'agente deve essere distribuita tra:

* codice sorgente;
* documentazione semantica;
* memoria del progetto;
* database vettoriale;
* grafo delle dipendenze;
* storico delle decisioni;
* stato operativo dei task.

La cartella `.ai` rappresenta la memoria persistente del progetto e deve essere presente all'interno del repository.

---

# Struttura principale

Esempio:

```text
project/

├── src/
├── tests/
├── docs/
├── README.md
├── .gitignore
│
└── .ai/
    │
    ├── project.md
    ├── rules.md
    ├── architecture.md
    ├── glossary.md
    ├── state.json
    │
    ├── decisions/
    │
    ├── tasks/
    │
    ├── knowledge/
    │
    ├── graph/
    │
    ├── vector/
    │
    ├── memory/
    │
    └── logs/
```

---

# Cartella `.ai`

## Scopo

La cartella `.ai` contiene tutte le informazioni necessarie affinché un agente possa lavorare sul progetto senza dover ricostruire ogni volta la conoscenza da zero.

La memoria deve essere:

* leggibile dall'uomo;
* versionabile;
* aggiornabile automaticamente;
* separata dagli artefatti temporanei.

---

# File principali

---

# project.md

## Scopo

Descrive il progetto ad alto livello.

Deve essere letto dall'agente all'avvio.

Contiene:

* nome progetto;
* obiettivo;
* stack tecnologico;
* vincoli principali;
* requisiti fondamentali.

Esempio:

```markdown
# Project

Nome:
Sistema gestione ordini

Stack:

Backend:
FastAPI

Frontend:
React

Database:
PostgreSQL

Vincoli:

- API REST
- Docker obbligatorio
- Test automatici richiesti
```

---

# rules.md

## Scopo

Definisce le regole obbligatorie che l'agente deve rispettare.

Contiene:

* coding standard;
* regole architetturali;
* convenzioni;
* limiti operativi.

Esempio:

```markdown
# Coding Rules

## Funzioni

Ogni funzione deve avere:

- descrizione;
- parametri;
- valore restituito;
- gestione errori.

## Architettura

- Separare business logic e controller.
- Utilizzare dependency injection.
- Evitare codice duplicato.

## Testing

Ogni nuova funzionalità deve avere test.
```

---

# architecture.md

## Scopo

Descrive la struttura architetturale corrente.

Contiene:

* componenti principali;
* comunicazioni;
* pattern utilizzati;
* decisioni architetturali attuali.

Esempio:

```markdown
# Architecture

Frontend:

React

Backend:

FastAPI

Database:

PostgreSQL

Pattern:

Repository Pattern

Authentication:

JWT
```

Questo documento viene aggiornato quando cambia l'architettura.

---

# glossary.md

## Scopo

Creare una conoscenza semantica del codice.

Ogni funzione, classe o modulo importante deve essere indicizzato.

Esempio:

```markdown
# UserService

Responsabilità:

Gestione operazioni utente.

## create_user()

Descrizione:

Crea un nuovo utente.

Input:

- UserDTO

Output:

- User

Dipendenze:

- UserRepository

Impatto:

MEDIUM

Ultima modifica:

Task 034
```

---

# state.json

## Scopo

Contiene lo stato corrente del progetto.

Esempio:

```json
{
    "project_phase": "development",
    "active_task": 42,
    "architecture_version": 3,
    "last_analysis": "2026-07-11",
    "files_indexed": 1250,
    "functions_indexed": 8400
}
```

Viene utilizzato all'avvio dell'agente.

---

# decisions/

## Scopo

Memorizzare le decisioni architetturali.

Formato:

Architecture Decision Record (ADR).

Esempio:

```text
decisions/

ADR-001-database-choice.md
ADR-002-authentication.md
ADR-003-api-versioning.md
```

Ogni decisione contiene:

* problema;
* alternative valutate;
* scelta effettuata;
* motivazione;
* impatto.

---

# tasks/

## Scopo

Gestire il ciclo di vita delle modifiche.

Struttura:

```text
tasks/

├── active/

│   └── task_045/

│       ├── request.md
│       ├── plan.yaml
│       ├── impact.md
│       ├── changes.diff
│       ├── tests.json
│       └── review.md

└── completed/
```

---

## request.md

Contiene la richiesta originale dell'utente.

Esempio:

```markdown
Aggiungere autenticazione tramite Google OAuth.
```

---

## plan.yaml

Creato dal Planner Agent.

Esempio:

```yaml
task:
  name: Google OAuth

steps:

- create oauth service
- update user model
- add callback endpoint
- create tests

risk:
  medium

affected_files:
  - auth/oauth.py
  - models/user.py
```

---

# knowledge/

Contiene informazioni strutturate generate automaticamente.

Esempio:

```text
knowledge/

├── modules.yaml
├── functions.yaml
├── classes.yaml
└── impact_map.json
```

---

# graph/

Contiene il modello delle relazioni del codice.

Utilizzato per:

* analisi impatto;
* navigazione;
* dipendenze.

Esempio:

```json
{
 "authenticate": {
    "calls": [
        "validate_password",
        "create_token"
    ]
 }
}
```

---

# vector/

Contiene l'indice del database vettoriale.

Utilizzato per:

* ricerca semantica;
* recupero codice;
* recupero documentazione.

Non deve essere necessariamente versionato.

Da aggiungere al `.gitignore`:

```text
.ai/vector/
```

---

# memory/

Divisa in:

## short_term.md

Memoria del task corrente.

Contiene:

* ragionamenti temporanei;
* file analizzati;
* problemi aperti.

Viene cancellata o archiviata a fine task.

---

## context.md

Contesto persistente operativo.

Contiene:

* informazioni utili recenti;
* stato generale;
* informazioni non strutturali.

---

# logs/

Contiene la tracciabilità completa.

Esempio:

```text
logs/

2026-07-11/

task_045/

├── planner.json
├── retrieval.json
├── agent_trace.json
├── test_results.json
└── final_report.md
```

Utilizzato anche per LangSmith.

# Versionamento Git

## Da versionare

```text
.ai/

project.md
rules.md
architecture.md
glossary.md
decisions/
tasks/completed/
```

Motivo:

Sono parte della conoscenza del progetto.

---

## Da ignorare

```text
.ai/vector/
.ai/logs/
.ai/cache/
```

Motivo:

Sono dati rigenerabili.

---

# Ciclo di aggiornamento memoria

La memoria non viene aggiornata durante una modifica non validata.

Flusso:

```text
Modifica codice

↓

Esecuzione test

↓

Review

↓

Approvazione

↓

Aggiornamento:

- glossary.md
- graph
- vector DB
- architecture.md

↓

Commit Git
```

---

# Principio fondamentale

La cartella `.ai` non deve essere considerata una semplice documentazione.

È il sistema cognitivo del progetto.

Il codice rappresenta cosa fa il software.

La cartella `.ai` rappresenta ciò che il software significa e come deve evolvere.
