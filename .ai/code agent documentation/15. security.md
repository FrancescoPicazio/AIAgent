# AI Software Engineer Agent - Parte 15

# Security Layer: sandboxing, permessi, rollback e controllo dell'agente

## Obiettivo

Definire il sistema di sicurezza che permette all'agente AI di modificare ed eseguire codice senza compromettere il sistema operativo o il progetto.

Un agente software autonomo ha capacità elevate:

* legge file;
* modifica codice;
* esegue comandi;
* installa dipendenze;
* interagisce con Git;
* avvia servizi.

Queste capacità devono essere controllate.

Principio:

```text id="m7p2x8"
Autonomia

+

Controllo

+

Tracciabilità

=

Agente affidabile
```

---

# Problema principale

Un LLM non comprende realmente le conseguenze delle proprie azioni.

Esempio:

Richiesta:

```text id="q8v3n1"
"Ripulisci il progetto"
```

Possibile interpretazione errata:

```bash id="c4m9k2"
rm -rf project/*
```

Risultato:

Progetto cancellato.

# 1. Permission Manager

## Obiettivo

Controllare cosa può fare ogni agente.

Ogni agente ha un livello di autorizzazione.

---

Esempio:

```yaml id="n5k8p3"

   read:
   true

   false

   false

   write:
   true

   execute:
   false



 tester:

   read:
   true

   write:
   false

   true
```

# 2. Capability-Based Security

L'agente non riceve accesso globale.

Riceve capacità specifiche.

---

Esempio:

Coding Agent:

Può:

```text id="v4q7m2"
✓ leggere src/

✓ modificare src/

✓ creare test
```

Non può:

```text id="p9x3k6"
✗ leggere password

✗ modificare .env

✗ cancellare repository
```

# Principio finale

Un agente che sa programmare ma può distruggere il progetto non è un software engineer.

È un processo automatico pericoloso.

L'obiettivo è creare un sistema con:

```text id="m9q3x6"

Intelligenza

+

Limitazioni

+

Verifica

+

Rollback

```

Solo questa combinazione permette vera autonomia.
