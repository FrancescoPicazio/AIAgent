# AI Software Engineer Agent - Parte 27

# Tool System e Agent Runtime: utilizzo controllato di terminale, editor, database e strumenti esterni

## Obiettivo

Definire il sistema che permette agli agenti di interagire realmente con il computer e con il progetto.

Un LLM da solo può:

* ragionare;
* generare testo;
* proporre codice.

Ma non può realmente:

* leggere file;
* eseguire programmi;
* compilare;
* fare commit;
* analizzare errori;
* interrogare database.

Serve quindi un livello operativo.

---

# Principio fondamentale

L'LLM non deve avere accesso diretto al sistema.

Architettura corretta:

```text id="m7q3x8"

LLM

 |

Tool Router

 |

Permission Layer

 |

Tools

 |

Operating System

```

# 1. Tool Manager

## Responsabilità

Gestisce tutti gli strumenti disponibili.

Funzioni:

* registrazione tool;
* selezione tool;
* validazione parametri;
* controllo permessi;
* logging.

---

Esempio:

L'agente decide:

```text id="x5m9p1"

Devo modificare auth.py

```

Non esegue:

```python
open("auth.py")
```

Direttamente.

---

Passa:

```python
tool_call(
"name":
"read_file",

"path":
"auth.py"
)
```

# Principio finale

Un agente senza tool è un consulente.

Un agente con tool controllati diventa un collaboratore.

La differenza tra un chatbot e un AI Software Engineer è la capacità di:

```text id="m7q3x9"

Pensare

↓

Agire

↓

Osservare

↓

Correggere

```

in un ambiente reale.
