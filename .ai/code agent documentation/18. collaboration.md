# AI Software Engineer Agent - Parte 18

# Multi-Agent Collaboration: coordinamento, conflitti e consenso tra agenti

## Obiettivo

Definire come diversi agenti specializzati collaborano per sviluppare software mantenendo coerenza.

Un sistema multi-agent non deve essere visto come tanti chatbot indipendenti.

Deve funzionare come un team software:

```text id="n8m4q2"
Architect

↓

decide struttura


Planner

↓

organizza lavoro


Coder

↓

implementa


Reviewer

↓

critica


Tester

↓

verifica
```

---

# Problema della collaborazione multi-agent

Aggiungere più agenti non aumenta automaticamente la qualità.

Possibili problemi:

* decisioni contraddittorie;
* loop infiniti;
* duplicazione del lavoro;
* agenti che modificano decisioni precedenti;
* aumento inutile dei costi computazionali.

---

# Principio fondamentale

Gli agenti non devono avere la stessa autorità.

Serve una gerarchia.

```text id="x7q3m9"

                 Supervisor

                     |

        +------------+------------+

        |            |            |

   Architect     Planner       Security

        |

      Coder

        |

    Reviewer

        |

     Tester

```

---

# 1. Supervisor Agent

## Ruolo

È il project manager virtuale.

Non scrive codice.

Gestisce:

* ordine attività;
* assegnazione agenti;
* risoluzione conflitti;
* stato progetto.

---

Input:

```text id="m4q8n2"
Feature request
```

Output:

```yaml id="p7m3x5"
workflow:

1:
Architect analysis

2:
Planner tasks

3:
Coding

4:
Review

5:
Testing
```

---

# 2. Agent Contract

Ogni agente deve avere un contratto.

Definisce:

* cosa può fare;
* cosa produce;
* cosa non deve fare.

---

Esempio:

```yaml id="q9m2v6"
agent:

Coder


can:

- modify files
- create tests


cannot:

- change architecture
- add dependencies without approval


output:

CodePatch
```

---

# 3. Shared State

Gli agenti non comunicano tramite chat libera.

Usano uno stato condiviso.

Esempio:

```python id="x4m8p7"
AgentState:

task

decision

files_changed

risks

tests

approval
```

---

Vantaggi:

* meno ambiguità;
* facile debugging;
* tracciabilità.

---

Caso comune.

```text id="m7q4x2"

```

---

Architect rileva:

```text id="n9m3p5"

```

---

Risoluzione:

```text id="q6x8m1"

HIGH

REJECTED
```

# 6. Reviewer vs Coder Conflict

Coder:

```text id="p3m8x6"

```

Reviewer:

```text id="v7q2n4"

```

---

Il codice non viene accettato.

```text id="k8m5q3"
"Passare i test non significa avere buon codice."
```

# Principio finale

Un team di sviluppatori non funziona perché tutti scrivono codice.

Funziona perché ogni ruolo ha:

* responsabilità;
* limiti;
* criteri di valutazione;
* comunicazione strutturata.

Un AI Software Engineer efficace deve replicare questa organizzazione, non semplicemente moltiplicare il numero di modelli.
